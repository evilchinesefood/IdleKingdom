# IdleKingdom UI Re-platform — Feasibility Critique

I now have complete evidence. I've verified the renderer, snapshot, solver, GraphView/GraphInput, the panels, the test harness (256 tests passing), the probe shim, and the FakeEl `addEventListener` behavior. I have everything needed to write the adversarial review.

Key codebase findings that drive my verdict:
- `fmtCost` (used by NodeInspector/HeroPanel) embeds `CURRENCY_GLYPH` emoji into button text — the design only flagged `Format/Format.js` in passing for P1, but the grep gate would catch it and the emoji is in a *string return*, not a vnode. This is a real wrinkle.
- GraphInput binds `pointerdown` on `this.svgEl`; link paths live in `layerLinks` (a child of svgEl). A click on a link will fire BOTH the link's `onclick` AND svgEl's pointerdown→`_down`→`onSelect(null)`. The design's "direct onclick on link-hit" will be immediately clobbered by `_select(null)` clearing `selectedLinkId`. This is a real interaction bug not addressed.
- Both FakeEl shims store only ONE listener per event type (`this._listeners[t] = fn`, overwrite). The design's P2 unit-test claim "assert `_listeners` count stays 1 / a removal spy fired" cannot detect listener *stacking* because the shim physically cannot stack — it overwrites. The leak-detection test as described is impossible under the shim.

# Adversarial Review: IdleKingdom WA Re-Platform Design

Verified against the live codebase on `main` (256 tests passing, not "248+" — minor staleness, harmless). All file/line claims in the design were checked against actual source.

## BLOCKERS

**B1 — Link click-to-reveal will be immediately clobbered by the canvas pan handler (interaction bug, item #3).**
`GraphInput._bind()` attaches `pointerdown` to `this.svgEl` (GraphInput.js:34). Link paths are appended to `this.layerLinks`, a child of `svgEl` (GraphView.js:37, 268). A pointer press on a link path therefore fires **both** the proposed `link-hit` `onclick` **and** bubbles to `svgEl`'s `pointerdown` → `_down()` → no port/node hit → `this.cb.onSelect(null)` (GraphInput.js:78-79) → GraphView `_select(null)`. The Render/Engine design explicitly says to *clear* `selectedLinkId` in `_select` — so the very same gesture that reveals a link's label will null it on the same event. Net effect: labels never appear.
*Fix:* The link reveal must be routed through `GraphInput`'s hit-testing (like `hitPort`/`hitNode`), not a bare SVG `onclick`. Add a `hitLink(gx,gy)` callback checked in `_down` **before** the `onSelect(null)` fall-through, and only toggle `selectedLinkId` on a tap (moved ≤ `TAP_MOVE_PX`), so pan-drags starting on a link don't toggle it. The "lowest-risk direct onclick" path the Render design recommends is in fact the broken one; the COMPONENTS-section instinct ("route link hit-testing through GraphInput._down like ports") is correct and should be the primary plan, not the fallback. Note: `pointerdown` doesn't even emit `click` reliably under pointer capture (`setPointerCapture` is called on every `_down`, GraphInput.js:44), so `onclick` on SVG children is doubly unreliable here.

**B2 — `fmtCost()` emits emoji inside returned strings; the P1 emoji-swap plan doesn't actually remove it, and the proposed grep gate will fail the phase.**
`Source/UI/Format/Format.js:6` defines `CURRENCY_GLYPH = { gold:"🪙", research:"📜", renown:"🛡️" }` and `fmtCost()` (line ~38) returns `"<n> 🪙"`. This is consumed by NodeInspector (`Upgrade → ${fmtCost(...)}`) and HeroPanel (`Level Up → ${fmtCost(...)}`, `Recruit → ...`). Because the glyph is baked into a **string return value**, it cannot be replaced by an `icon()` *vnode* the way the design assumes for HUD/ExpeditionBoard. The P1 plan lists `Format/Format.js` only parenthetically ("the `CURRENCY_GLYPH`/`fmtCost` path") and the FOUNDATION section's file list omits it entirely. Meanwhile the P1 acceptance criterion is a hard grep gate (`grep -rnP <emoji> Source/UI` returns nothing). That gate will **fail** on `Format/Format.js` unless `fmtCost` is restructured.
*Fix:* Either (a) change `fmtCost` to return text-only (`"<n> gold"`) and have call-sites prepend `icon("gold")` as a sibling vnode, or (b) explicitly exempt `Format/Format.js` from the grep gate in P1 and defer its de-emoji to P3 when NodeInspector/HeroPanel migrate. Pick (a) — it's the only one consistent with "no emoji remain." This is a small but real scope item the plan under-specifies.

## MAJOR

**M1 — The "no listener stacking" unit test (P2) is unprovable under both FakeEl shims; the leak risk is asserted-away, not tested.**
Both shims implement `addEventListener(t, fn){ this._listeners[t] = fn }` (PlaythroughProbe.mjs:94, Dom.Test.js:49) — a single-slot overwrite, not an array. The PHASING section promises a Dom.Test.js case asserting "`_listeners` count stays 1 / a removal spy fired" to catch handler accumulation from the 2s HUD re-render. Under a shim that physically cannot hold more than one listener per type, this test can never observe stacking and gives false confidence. The actual leak risk (the RENDER/ENGINE section correctly identifies it: re-rendering with a *new closure each frame* thrashes add/remove) is real in the *browser* but invisible to the test.
*Fix:* (1) Upgrade the test FakeEl's `addEventListener`/`removeEventListener` to maintain an array per type and expose a count, so the bookkeeping (`el.__waEvents` remove-before-add) is genuinely exercised; (2) keep this strictly a unit concern and ALSO add it to the per-phase browser checklist with DevTools "Event Listeners" inspection. As written, the test is theater.

**M2 — `wa-select`/`wa-input` `change` event behavior is unverifiable under the probe, yet P3/P4 migrate the most interaction-heavy controls to it.** The probe drives selects by directly invoking `sel.onchange({target:{value}})` (PlaythroughProbe.mjs:849, 1042, 1444). For a real `wa-select`, the value lives on the upgraded element and `change`'s `event.target.value` semantics depend on shadow-DOM upgrade that the shim never performs. The design acknowledges "can't render WA under node" generally, but the *specific* risk — that `e.target.value` on a `wa-select` `change` event may not be the selected `wa-option`'s `value` the way native `<select>` is — is the single highest-likelihood functional regression (recipe/gatherer/equip all ride on it) and is **only** covered by the manual browser pass. That's acceptable *if* the browser pass for P3/P4 explicitly tests "selecting option X dispatches intent with value X," but the checklists say "change a recipe, confirm node rebuilds" — they confirm re-render, not that the *correct value* propagated.
*Fix:* Make the P3/P4 browser checklists assert the dispatched intent payload value (e.g. via a temporary console.log of `dispatch.last`), not just visual re-render. Also pin: per WA docs the value-bearing event for `wa-select` is standard `change` with `event.target.value` — confirm this empirically in P2's throwaway probe before committing BuildMenu/NodeInspector to it.

**M3 — `prop:value` "skip if unchanged" guard can desync a `wa-select` after the user changes it, because the snapshot round-trips a stale value on the next intent render.** The render cadence renders panels on *every* intent (App.js:132 `_onSnapshot`→`_renderScreen`). When a user picks recipe B, the `change` handler dispatches `SetRecipe`, which produces a new snapshot and re-renders NodeInspector with `prop:value = node.recipeId`. That's fine when the engine accepted it. But the `if (el[name] !== v) el[name] = v` guard (RENDER/ENGINE A2) compares against the element's *current* property — if WA's `change` fired optimistically before the component's `value` reflected, or if the intent is rejected (Reducer returns `{ok:false}` — see the probe's `assert(r.ok,...)` pattern), the panel re-renders with the *old* `recipeId` and the guard happily re-asserts the stale value, silently reverting the user's visible selection. The design treats `prop:value` as purely defensive; it can be actively wrong on rejected intents.
*Fix:* Always keep selects **keyed** (the design does say this — `key:"recipe-"+node.id`) AND drop the `!==` guard's role as correctness-bearing; rely on keying for stability and let `value` always reflect the authoritative snapshot. Verify reducer-reject behavior for `SetRecipe`/`SetGathererResource`/`EquipItem` in the browser pass (pick an invalid option, confirm the select snaps back coherently).

**M4 — Buildless WA loader + FA webfont vendoring is sound, but two version/packaging facts are unconfirmed and gate all of P1.** The research itself flags (correctly) that (a) `dist-cdn/` may be absent from some packaged versions (GH #2146), and (b) the exact `--wa-color-*` neutral-scale slot names are version-sensitive. The design's mitigation ("grep the vendored CSS after install") is right but is a *post-install discovery* — meaning P1's WaTheme.css token list is **speculative until the package is actually pulled**. There is genuine risk the token names (`--wa-color-brand-fill-loud`, `--wa-color-surface-raised`, `--wa-border-radius-m`) don't exist in v3.7.0 and the theme silently no-ops (parchment/iron/gold won't apply, components render in default blue).
*Fix:* Sequence P1 as: pull package FIRST → `grep --wa-color- webawesome.css` → THEN author WaTheme.css against confirmed names. Treat the token list in the design as a draft. This is a process fix, not a design flaw, but the plan presents the token file as ready-to-write.

## MINOR

**m1 — Test count drift.** Design says "248+ tests"; actual is **256 passing, 0 failing**. Harmless, but update the invariant so the regression gate has a real number.

**m2 — `atCapacity`/`starved` derivation is correct and well-grounded — confirmed.** The RENDER/ENGINE design's handling of the scholar/market consumer case (their `availableOut` is `{}` per RateSolver.js:99,122, so `effectiveRate`=0; use `perNodeDraw` instead) is accurate and is the *right* catch — without it, every fully-fed scholar/market would falsely read "starved." The solver already returns `capacityByNode`, `availableOut`, `perNodeDraw` (RateSolver.js:158-165). No solver change needed; claim verified. One nuance the design gets right and the COMPONENTS section gets subtly wrong: COMPONENTS says "gatherers always run at capacity (no inputs) so they're always MAX" — but a gatherer with `resourceId=null` has `availableOut={}` (RateSolver.js:78) and `effectiveRate=0`; the RENDER/ENGINE `atCapacity = cap>0 && throughput>=cap-EPS` plus `takesInput=false` correctly yields "neither badge" for an unconfigured gatherer (cap may be >0 but throughput 0 → not atCapacity, not starved since `takesInput` false). The two sections disagree; **RENDER/ENGINE's formula is the correct one** — make it canonical and discard COMPONENTS' "gatherers always MAX" shorthand.

**m3 — `wa-tab-group` as "styled tab bar only" (no `wa-tab-panel`) is viable but undertested.** The design keeps the hash Router authoritative and drives it from `wa-tab-show`. WA tab-groups manage their own internal active state on click; with no `wa-tab-panel`s present, confirm the group still renders all `wa-tab`s and fires `wa-tab-show` (some tab implementations require matching panels). Low risk, but put it on P2's browser checklist explicitly.

**m4 — ResearchTree SVG edge passthrough (`{el}` vnode) interaction with WA cards is fine — confirmed.** `isPassthrough` (Dom.js:21) returns the prebuilt node as-is; mixing it with `wa-card` siblings in the same `patch()` works since custom elements are ordinary Elements. No issue.

**m5 — `user-select:none` on `wa-*` host selectors may not reach slotted/shadow text.** The COMPONENTS section notes this; correct. The rule on `wa-button`, `wa-tab`, etc. styles the host but WA components render label text into shadow DOM / slots. For chrome this is fine (labels are short), but verify the dialog/callout body text (which the design *wants* selectable via `wa-input, .modal-text, .victory-text { user-select:text }`) actually honors it through the dialog's slot. Browser-pass item.

**m6 — Mixed-phase transition is genuinely safe — confirmed.** The reconciler extension (P2) is purely additive (`onWa*`/`prop:` are new prefixes; `on*`/attribute paths at Dom.js:36-55 are untouched), so un-migrated panels render identically. App.js's per-route `patch()` calls (157-169) are independent hosts, so a P2-migrated HUD coexists with P1-era native panels. Each phase's rollback story holds.

## Scope / Risk

- **Beta-library risk is real but bounded.** WA v3.x token rosters and the standard-vs-`wa-`-prefixed event split are the two live API hazards; both are correctly flagged. The mitigation (vendored, pinned, single WaTheme.css tuning point, browser pass per phase) is the right posture. The biggest *unflagged* risk is **shadow-DOM focus/keyboard behavior in `wa-dialog`/`wa-select` that the node probe is structurally blind to** — P3 and P5 lean entirely on the manual pass for these. That's honest, but the per-phase checklists need the *specific* assertions (M2, M3) added or the "automated safety net" overstates its reach.
- **Buildless-vendored vs tiny build:** no reason to reconsider. FA webfont `<i>` is the only offline-clean duotone path (kit-code = runtime CDN, `registerIconLibrary` Pro-duotone unsupported), and WA `dist-cdn` is purpose-built for this. The `<i>`-not-`<wa-icon>` decision is well-justified and keeps the renderer trivially compatible.
- **SVG node-card icon (P3) remains genuinely unsolved.** All four sections waffle (foreignObject vs monochrome glyph vs unicode codepoint). FA duotone's two-layer `::before/::after` cannot render in SVG `<text>` — confirmed constraint. `<foreignObject>` is the only full-duotone option and it works, but flag that `<foreignObject>` re-introduces HTML layout inside the hand-rolled SVG that the `_replace()` teardown (GraphView.js:348) must handle. Not a blocker (P3 decision), but stop calling it "decided" — it isn't.

## Verdict: **FIX-FIRST — 2 must-fix (BLOCKERS) before P1/P3 implementation.**

- **B1** (link-reveal clobbered by pan handler) must be redesigned through `GraphInput` hit-testing before P3 — the headline item #3 does not work as drafted.
- **B2** (`fmtCost` emoji vs the P1 grep gate) must be resolved before P1 can pass its own acceptance criterion.

The four MAJORs (M1 unprovable leak test, M2/M3 select-value verification gaps, M4 token-name sequencing) are not blockers but will each surface as a "passed CI, broke in browser" surprise if not addressed; fold them into the phase plans before starting the affected phase. The core architecture — additive reconciler extension, snapshot-derived `atCapacity`/`starved` (fully derivable, verified), keep-the-canvas, vendored-buildless delivery, phase independence — is **sound and confirmed against the code.**
